# Project profile — WaitlistKit

## 1. Identity & framing

- **What this product is:** A hosted viral-waitlist SaaS — founders create a coming-soon page in minutes; every subscriber gets a unique referral link that improves their queue position as friends join (Robinhood/Superhuman-style growth, no code required).
- **Framing discipline (words we own / avoid):**
  - Own: "queue position", "referral link", "get early access", "waitlist", "launch"
  - Avoid: "newsletter", "lead capture", "revolutionary", "game-changing", dark-pattern urgency ("only 5 spots left!" when false)
- **North-star metric(s):** Viral K-factor (avg verified referrals per signup); Free → Paid conversion rate (target 3–6%)

## 4. Product Rules (the testable rule set)

> QA names tests against these IDs (e.g. `test-REF-1.spec.ts`). Reviewers check coverage.

| Prefix | Domain | Rules |
| :----- | :----- | :---- |
| REF | Referral engine | REF-1: Every signup receives exactly one unique referral code · REF-2: Self-referral is rejected (same email = same person) · REF-3: Position score is recomputed immediately on each new verified referral |
| ANTI | Anti-fraud | ANTI-1: A signup's position improvement is only counted after email is verified (double opt-in) · ANTI-2: Duplicate email submissions to the same waitlist are rejected with a 409 |
| BILL | Billing / plan gates | BILL-1: Signup count cap is enforced per plan (Free ≤ 100, Starter ≤ 2000, Pro ≤ 25000) · BILL-2: WaitlistKit branding removal requires Starter+ · BILL-3: Custom domain requires Pro |
| DATA | Data / PII | DATA-1: Subscriber email is persisted only after explicit submission of the opt-in form · DATA-2: CSV export of subscriber data is accessible only to the authenticated waitlist owner · DATA-3: All subscriber data is purged on waitlist deletion |
| PAGE | Page builder / hosting | PAGE-1: Public waitlist page (`/[slug]`) is accessible without authentication · PAGE-2: Waitlist slug must be globally unique; a conflict returns a 409 |
| AUTH | Auth / access control | AUTH-1: All founder dashboard routes require an active Clerk session · AUTH-2: A founder can only read/write/delete their own waitlists (no cross-account access) |

## 5. Safety surfaces (what triggers the mandatory gate)

> A change touching any of these surfaces requires `safety-reviewer` mandatory review with **veto**.

- **PII — subscriber emails:** Collected, stored, and used for opt-in + automations. Any change to the email capture form, storage logic, Resend integration, or unsubscribe path is a **safety surface**.
- **Payments — Stripe webhooks + plan gating:** Billing integrity (plan limits, charge events, webhook signature verification). Any change to `/api/webhooks/stripe` or plan enforcement is a **safety surface**.
- **Auth / access control:** Clerk session checks and owner-isolation (AUTH-2). Any change to middleware or ownership predicates is a **safety surface**.

No minors-specific surfaces declared. No AI/LLM output to end users. No health data.

→ **Safety gate mode:** mandatory review on PII / auth / billing surface changes (`safety-reviewer` holds veto); advisory on all other changes. The gate is **never silently dropped**.
